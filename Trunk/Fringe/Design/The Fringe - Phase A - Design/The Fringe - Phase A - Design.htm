Design Summaries

Project/Namespace Architecture

 

Some Notes Regarding the Project/Namespace Architecture Diagram:

  • To make the hierarchy more clear, I did not make any connectors for lower level libraries when a connector was already made.  For example, Framework.UI.dll would connect to OSG, but I did not need to make that connection explicit, because it was already connected to Framework.UI.OSG.dll.  The only places I broke that rule was in the Fringe Clients.  Fringe.Client.exe and Fringe.Server.exe both make the primary connection to GG_Framework.UI.dll because that is where they make the OSG window.  All three clients make the primary connection to GG_Framework.Logic.Network.dll, because they are setting up the Client and the Server.  GG_Framework.Logic.dll also makes these connections to the UI and Network, but the dependency is much shorter, just so it can set up all of the logic.
  • Italicized items represent 3rd party components, in the Source/Utilities folder.  The two osg plugins for lwo and lws files are not italicized, even though they are third party, because we actually do touch their source.
  • Lighter colors are higher in the project abstraction chain, and darker colors are lower down.
  • The vast majority of the classes are found in the framework, which means they are going to be reusable for future games.  Within the Fringe itself are the EXE’s and a library, Fringe.Designer.dll that Justin can work on directly to put very Fringe specific stuff.  It is based on another library that will provide base classes for Justin’s work, called Fringe.Base.dll.
  • Colors represent certain types of libraries, based on the component diagrams shown above.  These libraries colors will be used in the static class diagrams to further associate classes with the library/namespace where they are defined.
    • Purple for UI
    • Blue/Aqua for Logic
    • Green for Network
    • Red for Base Framework
    • Yellow/Tan for Fringe clients
    • Grey for OSGV
  • EXE projects are intentionally very thin.  For OSGV, this was because I wanted to apply a high-level automated tester to the library.  For the Fringe Demo, the underlying client and server libraries are re-used across the various libraries.  The main() function in the EXE will set up all of the main instances and will run the game loop and manage the Timer.
  • You might notice that there are some deviations from how we currently have the libraries built.  I will be taking the code and placing it in the repository according to this architecture.
    • I am going to use the slight renaming convention (and spelling correction) “GG_Framework” vs. “FrameWork” to represent the newer framework libraries.  We will keep the deprecated FrameWork library around for a little while until everything is copied over.
    • I merged the Audio.SoundEffects3D in to the single Audio library/namespace.  It follows the flow of how the other stuff works better, and I do not feel that the Audio library warrants such a deep structure.  When we get music involved, it can go into the Audio library as well.  In the end, I am sure there will be less than a dozen or so classes, and that is plenty small enough for a library.
    • The GG_Framework.UI.dll library corresponds to what is now FrameWork.ActorActions.dll.

Actors

Future versions of the Fringe, and every other game we make, will have other actors besides the ones listed here, including technical support staff.  For Phase A, we have these actors:

Primary Actor – Player

This is of course the primary actor of the game.  He will be playing the game, probably the Single version, but he might also be playing the multi-player client.  See Advanced User and Simple User below.  Advanced and Simple Users are only distinguished in the Summary Use cases.  In the Feature Use cases, both the Simple User and the Advanced User will just be considered Players.

Primary Actor – Advanced User : Player

The Advanced User is most likely someone from the development team.  They can do everything that the Player can, but he might also instantiate the Server with a scene script and play the multi-player version of the game with multiple clients.

Primary Actor – Simple User : Player

The Simple User could be a potential investor or anyone else we want to show the game off to that is not a part of the development team.  They will not know how to set up a server/client arrangement or how to specify a script to run.  They will merely play the Fringe.Single.exe, without wanting to know about any parameters.

Secondary Actor – Designer

The designer is considered an actor, meaning outside of the system, because he is acting on the system to write scripts and callback code to show off his vision.  I might use the name “Justin” to represent the Designer interchangeably.

Components

The use cases will essentially written like typical system requirement use cases, except that since we are working for lower level design, rather than just requirements mining or presentation.  I want to describe the “System” in more detail, so the interactions in the use case will include actors and the following components.  The components described will then correspond to the various layers listed in the Component Design. There are also these components:

Game Timer

This represents the main timer, that starts with the Server, but is synced through the Server to each of the Clients.

Game Script

The Script File that defines the entire game for all of the players.

LWS Plugin & LWO Plugin

There are some features that deal specifically with the LWS and LWO plugins for OSG.

Server / Client

These terms are often used when a connection is made between the applications.  It infers a communication between the Network Layers. 

OSG Window

The primary window provided to Player.

Logo Scene

A quickly loading scene that is displayed while the game content is being loaded

Server Game

The representation of the game on the Server, including the AI Networking Layer, AI Logic Layer, and all AI Controllers.

Client Game

The representation of the game on the Client, including the UI Layer, the Player Logic Layer, and all of the Controllers.

Game Content

“Game Content” is all of the content (osgv, lwo, lws, audio) files for all of the existing Entity3D objects that is sent from the Server to the Client when the Client connects.  This content is used to create the Client Game.

  4. EXE reads Game Script to create AI Logic (Notify & Abort on Error)

EXE reads Game Script to create AI Network and AI Logic

Be sure to Notify via the console and abort on error.  The game designer is going to need to create a script that defines the game.  Designers must be able to include this information in the script:

  1. Each AI Entity3D in the scene (including window dressing items, just uses a simple AI controller)
  2. When a new player comes in, we need to know about what ship he is flying (perhaps with a callback)
  3. Spawn Points (Callback for when a Player needs to spawn)
  4. Some way to provide information to all of the connected players, like when the primary target is destroyed, or when the player dies and there is no remaining spawn point.
  5. Music Sound Track

 

I am thinking we should use a LUA script for this, because we can make callbacks for the script.  The script itself can hold a kind of game logic and be notified of game events on the server side (like clients connecting, ships being destroyed, clients disconnecting, etc.)

 

When this file is read in successfully, the EXE will create the AI Logic based on all of the AI Entity3D objects in the scene.  If there is an error of any kind reading the file, we need to notify the user and abort.

The LUA Script Provides all of the Entity3D Information

The LUS script is going to need to provide allof the ship information for the AI Client as well as each of the other clients.  Some things to keep in mind:

  1. For each ship, we will want
    1. OSGV file (Represents the BB and UI version of the OSG scene graph)
    2. AI or UI Controller and parameters
    3. The type of Entity3D and parameters
    4. The type of Sensor and parameters
    5. Some kind of unique identifier for passing around information
  2. We will want to pre-load all of the possible extra objects, like missiles and such, even though they are not part of the actual scene.  Perhaps we need to keep references to these extra objects in some container when they are not in the actual scene.
  3. Only player clients need to read in the UI version of the osgv file
  4. Only the controlling peer needs the Controllers, Specialized Entity3D and Sensors
  5. I want to centralize the location of this stuff, perhaps by calling functions with parameters
  6. We will also need to provide Player PEERS with cosmetic enhancements

Client/Server Connections

Server accepts Client connections and provides Initial Game Content

When the server sees that a Client has connected, it starts a communication with that Client, it needs to run the Game Script for a new Client.  This will tell the Server what kind of Ship the Player will control, and it might even tell the player that there are too many connections of a given type and turn the player away.  If the Server determines that the Player logging on is valid, it can start getting the Client up-to-date on the game.  This is a process of communication that happens before the Client actually allows the Player to start controlling his ship.  If the connection from the Client is broken at any time during this update, we will need to let the Game Script know that the Client did not complete loading, and is no longer in the game.

 

The Server needs to make sure that the Client has a current representation of the initial Game Content, just enough to load all of the files it needs.  The Client accepts the Game Content and loads all of the files, and sends a confirmation message back to the Server that it is ready for the next communication.  Note that in this version, we are assuming that all of the files are already on the Client hard drive.  There may come a time that this is not true, and the maps and models for other players need to be sync’d.  This loading takes time for the Client, so the current state (like current position of all of the ships) does not need to be sent right away.  The content to be sent is really about all of the Entity3D objects that are in the current scene, including the AI ships that originated in the scene, as well as all of the other players.  Basically it is the content for each 3D Entity in the scene, as described in Server Notifies all Clients of new Player Entering the Game, but in all one packet.  All of these entities will need to be loaded with NETWORK Controllers.

 

It is possible, due to a relatively slow Client Connection or a relatively slow Client loading the files, that another Player may join the game while this Client is still loading the initial content.  If that happens, the Server should just go ahead and send the information about the 2nd Player to the 1st Player.  The 1st Player will not pull this new information off its Client Network queue until after it is done with all of its loading.  Then it can accept the content from the 2nd Player.

 

The Server needs to wait for the return from the client, saying that everything is loaded (including these other Players) before it can notify all the Clients about this new Player entering the Game.  As far as the Server is concerned, this should all be pretty fast, so it does not need to happen on a separate thread.  It is not sending a lot of information, mostly filenames for the Client to load.

(Client) EXE accepts Game Content from Player Network Layer and creates Client Game

While the logo is being displayed on a separate thread, the Client (or Single) EXE receives the game content from the Server.  It works through this content and controllers to create the Logic Layer, including all of the Entity3D instances with regular network controllers (since all of these Entity’s are being controlled purely from external sources, not from this Client).  It also creates the underlying nodes for OSG by creating an Actor and reading in the OSG file.  We also need to load the music tracks that are going to play in the background.  This load might take time.  Note that these nodes are not yet connected to the camera (it is still seeing the logo scene.  This load might take some time, so it should happen on another thread while the logo spins or whatever.  When the loading is complete, the Client sends a message back to the Server saying it is complete.  If there is a problem, we need to notify the Player and then exit the app (notify the spinning logo window status text), breaking the connection with the Server.

Server Notifies all Clients of new Player Entering the Game

When a new player joins the game, all of the other players need to know about it.  As a matter of fact, even that player needs to know about it.

  1. The VISIBLE files (osgv, lws, lwo, and image files).  Note that ALL files should be loaded for the ship, including all of its potential states (e.g. damaged) and what kinds of weapons and such it may spawn.
  2. Bounding box information and other logic points (e.g. target-able points)

 

It is for this reason (#2) that the logic layer needs to know something about OSGV.  The Designer will need to have a way to easily define the elements of the 3D model that are needed by the logic.  This will happen in their 3D package of choice (LightWave).  It may be that the bounding boxes are defined as some non-visible layer(s) in the LWO file, denoted by some naming convention.  Other points might be denoted with specially named NULL objects.  There may come a time when we use multiple layers in the LWO file to represent multiple levels of detail (LOD).

 

So all of the players will receive everything they need to know about the other player’s ship, just as if it were another AI ship.  The player that is starting the game will also need to know things specific to his controller.  This includes the class name that is to be instantiated and all of the art for the Player’s HUD and other user interface features.

 

As far as the Server is concerned, this should all be pretty fast, so it does not need to happen on a separate thread.  It is not sending a lot of information, mostly filenames for the Client to load.  The Server DOES need to wait for all of the other players to complete that load and return with a notification before it can let the player continue to play.

 

As for providing the bonding box representation of objects in the scene, we realized that we will need to use LightWave to create the models.  We can use the same OSGV file and LWS files to represent the UI versions of the ship as well as the Bounding Box (BB) representations of the ship.  The difference will be that for each .lwo file representing a ship, (e.g. X-wing.lwo), the designer will make another .lwo file that represents the bounding box.  It will have the same number of layers and configuration as the original file, and will be named with a convention (e.g. X-wing_BB.lwo).  The scene file that represents the ship with its articulations (e.g. X-wing.lws) will just reference the original file (X-wing.lwo).  The OSGV file (e.g. X-wing.osgv) representing the actor and actions will look at the lws file (X-wing.lws).  When the SceneManager code reads in the actor actions, it will read it in as it does now for the UI part.  When it reads in for the BB part, it will parse the same OSGV file, but this time, when it reaches a lws file (X-wing.lws), it will make a temporary copy of that file (e.g. X-wing_BB.lws) and modify it on the fly to replace all instances of the string “.lwo” with the string “_BB.lwo”.  It is this modified version of the file that is read with the osgDB_lwo plugin, so the nodes read in will be the bounding box versions of the .lwo files.

(Player) Accepts New Players

Whenever there is a new player, all of the content needs to be read in (osgv, lwo, lws, audio files).  Existing players are going to get this new info and read it on a separate (low priority) thread so as to not interfere with their game.  When they are done loading that information, they can send a message back to the Server saying they are done (watch for disconnects before being done).  The new player cannot start their game until all players that are still connected have finished loading all of their content.  See Server Notifies all Clients of new Player Entering the Game.  Remember that this content is not just the new ship, but any other content as well, like weapons they might fire, other states they might find themselves in (e.g. LOD, damage, etc.)

 

If the new player information coming in is about another Player or some other AI, then the new Controller will be a simple AI controller, and not too much needs to happen as far as the HUD or other on-screen elements.  If the new information being loaded is about the Player’s own ship, then it needs to create the appropriate UI Controller, which will then load all of the content (osgv, lwo, lws, audio) files associated with control aspects of the new ship.

Server starts the Game Timer

Up until the first Player is connected and finished loading all of the initial game content and content from its own ship, the Server does not need to be running its own game at all.  This essentially means that the timer has not started.  Once the Server gets the message that the first player has finished loading, it can start its own game Timer at 0 and start looping through the logic.  The timer is actually a double precision value that represents the Time on the Server.

Server notifies new Player of current Game State

By this time, all of the content has already been loaded and the Client is ready with the OSG and logic scenes ready to go, they just do not have good starting positions.  This initial data needs to include:

  1. The initial Timer value, and the Server system time.  RAKNET and the Client will resolve the difference between the Client System time and the Server System time to get an approximation on the Client machine of all timed messages.
  2. Positions and Orientations of all existing 3D Entities, (this includes missiles)
  3. State of all existing 3D Entities.  This includes state that needs to be shown on a HUD (like damage level), and the current Action the Actor is displaying, and when that current Action was fired.  Actions may need to be adjusted so that they can be fired at a time BEFORE the current time.  I think this will mostly work ok, except that there may be some kinds of effects that can’t handle that, like sound effects and particle effects.

Server accepts Quit (disconnected) Clients, Notifies other Clients

I do not think there needs to be a special message for when the Client quits.  RAKNET will automatically send a message that says that the client has disconnected, so I think this will be ok, certainly for now.  When one player does disconnect, the other players need to know about it, so the Server needs to tell them all.

(Player) Removes Quit Players

When a Player receives a message that another Player is out of the Game, it needs to remove that Entity3D and all of its associated Actors from the OSG and Physics scenes.  We also need to watch for targeting and other issues. I am not sure if the other Player should disappear, or perhaps it should explode, and just not re-spawn.  This would probably be a better way to go, since we have to handle the leaving player being removed from targeting lists and so forth.

Game Play Design

These Use Cases and Development Units are all about the interaction of the ships and other entities in game play.  I wanted to go into a little more detail with regards to how the UI and the Logic components interact, based on the Design Diagrams below.  We will describe the development units in relation to a specific game set up, which includes 2 Players (Shio_P1, Ship_P2), with their own ships and 1 additional ship controlled by AI (Ship_AI).  We will also include a Planet, Sun, and Star Field.

Use Cases

Player Plays Game

While the Player is playing the Game, any of these things can happen (not in order):

  1. Player Controls his own Ship (Tweak-able by the Designer)
  2. Player sees ships and entities controlled by other Players or AI
  3. AI Controls other AI Ships and Entities and respond to each other and Player’s ships (Scripted by the Designer)
  4. Player sees on-screen notifications from the Server and can cancel out of them
  5. Player’s Ship can be destroyed and Player is Re-spawned
  6. Player can crash into other entities (tight bounding boxes)
  7. Ships respond to external Forces
  8. AI can crash into other entities
  9. Player and AI ships can fire Weapons at each other
  10. Player can see the results of Actions on his own ship and other entities

Player Controls his own Ship

Ship control is extremely specific to the ship.  The UI layer is going to accept mouse and keyboard input.  That input will be passed to a Player Controller that will control not only what happens to the ship, but things that happen to the camera as well, like the chase-plane view or in-cockpit view.  There are also going to be HUD and other on-screen controls. Each of these things is very specific to the kind of ship being flown, so the Designer will need very tight control.

 

Here are some of the things the Player is going to need to be able to do while flying his ship:

  1. Launch from the Launch Ship
  2. Look around with the Mouse and KB
  3. Control the Ship with direct Mouse and KB Input, allow for at least one evasive maneuver.
  4. Target other Ships and Ship Components
  5. Lock on and Fire Missiles
  6. Fire the Plasma Gun
  7. See the Artificial Horizon Indicator

 

Development Notes

Weapons

  1. The Player’s Ship can fire missiles. 
  2. The Player’s Ship can fire a Plasma Cannon
  3. The big bad ship fires an AA Gun at the player
  4. Big Ships fire big cannons at each other

 

Missiles are represented as 3D objects, and trails are very important.  There should be several swarmed at a time and they can track with some degree of accuracy.  They might take a bit to load up and lock on.

 

The Player’s Cannon is like a BUZZ gun, appearing to have a very high firing rate.  It is actually going to be firing sprites.

 

The AA gun fires from the enemy’s ship cannons at the player.  They are probably sprites also with a slower rate of fire (easy to evade)

 

When big ships fire at each other, they are firing big, non-tracking weapons with huge warheads that do lots of damage.  These are probably going to be sprites as well, but they may need a tail, so they could possibly be 3D.

 

From an Engineering stand-point, these are all AI driven entities.  The AI may be driven from the Player’s client logic, OR from the Server’s logic.  Most importantly, we will need to SPAWN these objects dynamically and let them die in a certain amount of time after firing.  We should pre-load a representation of each weapon that can fire and keep it out of the scene graph so that first shot does not take load time.

Look around with the Mouse and KB

When the player is flying from the cockpit, he can see a forward view that sways a bit with his motion.  There needs to be an easy button press that will allow him to look back, left, and right.  The ship should show semi-transparent.

 

The player can zoom into and out of this in-cockpit view to more of a chase plane view, like in BFII, with an adjustable range (perhaps with the scroll wheel?).  Once he is out there, he should be able to press a key, disabling ship turning momentarily (but he should still be able to do other things, like fire, arm, break, etc.), so he can trackball around and see his gorgeous ship.

 

There are times when game logic (rather than the player) determines how he is looking at is ship, such as when he launches from another ship.  We should keep these view modes distinct.

Target other Ships and Ship Components

The player needs to have a set of targets, with a single active target.  He can cycle between the targets.  A target can be a ship or a target-able component on a ship that has not been destroyed.  Each target is represented by a bill-boarded image map that is always drawn on top and at the same size.  The non-active targets are dimmed out a bit, either using a separate image, or better yet a more transparent image.  There is an arrow shown on the border of the screen pointing to the active target.  There may also be some text displayed on-screen stating the current target.  Targets need to also show, with the image box, a name, distance, and a bar showing damage.  We will eventually need to let the player choose his targets by clicking and/or from a list, but for this demo, let’s just have all items in the scene be pre-targeted, and the player can cycle through them.

 

The Designer must be able to indicate target-able components of a ship in the LWS/LWO file, perhaps with NULL objects with a special naming convention, so we can use the target name as well.  The game-script also needs to provide a name for the ships so they can show a target name.

Artificial Horizon Indicator

This indicator is one of the only HUD items we have for the investor’s demo.  It might be an optional feature, so it should be done later in development.  It will be a 3D object, parented in such a way that it that roles along with the player.  There will be lines representing the horizon, with numbers on the edge that are parented and bill-boarded.  Justin wants the AHI to be more opaque in the middle of the screen and more transparent at the edges of the screen.  I am not sure exactly how that will work, since this transparency will be moving relative to the actual image map.  We may use image maps to do this, or perhaps we could draw lines and fade them.  The numbers would be drawn separately, perhaps as image maps, or maybe as actual text.  It seems this indicator should only show when in cockpit view, so it should fade out when not there (rather than snapping away).  We may also want to allow the player (or commander, or game logic) lock this artificial horizon indicator to other horizons, like the one for his flight group, or for the enemy flight group.

Launch from the Launch Ship

The player will always be spawning from a launch ship.  Rather than physically parenting the player’s ship in the scene, we will most certainly have a callback that listens to the changing position each frame of the launch ship and update the player’s ship position accordingly.  Eventually, all ships that need to re-spawn will start on a larger ship.  This means that the player may be waiting in line to launch.  Perhaps they should be able to watch the ships in front of them launch while they are waiting for their ship to materialize.  Perhaps there is a hatch bay that opens and reveals their ship.  This means that the view position needs to be set around where the launch position is, and not always where the player’s ship is (since their ship is not even on screen yet).  They may also be seeing some kind of information screen that they need to dismiss before launching, especially on the first launch.  Do they then need to press some button to launch, which catapults them and places them in control of their ship, OR perhaps the launch is timed and automatic, with cool artwork that builds towards the launch.  If the Launch ship has been destroyed, the Player sees a screen that says the game is over, and they can trackball around any remaining ship or something in the system until they quit the game.
